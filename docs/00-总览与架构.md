# 00 · 总览与架构

> 对应原文档：一、文档说明 ／ 二、需求理解与系统边界 ／ 三、总体技术方案 ／ 四、推荐项目目录结构 ／ 五、核心模块设计 ／ 十二、配置文件设计

本文件是全局设计基线，各阶段文档均以此为共同上下文。

## 1. 项目定位

| 项目 | 说明 |
|------|------|
| 项目名称 | 年轻球员潜力发现与报告生成系统 |
| 方案名称 | 方案 A：本地 Python 脚本版 MVP |
| 第一版目标 | 完成可运行、可解释、可展示的本地数据分析与报告生成流程 |
| 目标用户 | 学生项目团队、足球爱好者、模拟球探/分析师、老师或评审 |
| 技术难度定位 | 适合只学过一点 Python 的高中生逐步完成 |
| 联赛强度来源 | 优先参考 Opta Power Rankings，不把联赛强度作为决定性评分因素 |

**要解决的问题**：不是「预测年轻球员未来一定能达到什么高度」，而是在已有公开数据中，筛选出值得重点关注的 U21 边锋和中场球员，并用清楚、可解释的方式说明他们为什么值得关注。

## 2. 已确认核心需求

| 需求项 | 第一版决定 |
|--------|-----------|
| 年龄范围 | U21，即 21 岁及以下球员 |
| 位置范围 | 边锋 + 中场；边锋包括 LW、RW、LM、RM、Winger；中场按 CM、AM、DM 分开评分 |
| 最低出场时间 | 600 分钟，低于该阈值默认不进入正式评分 |
| 样本规模 | 至少输出 100 名符合要求的年轻球员基础样本清单 |
| 排行榜规模 | 输出 Top 20 潜力排行榜 |
| 报告形式 | 每名 Top 20 球员单独生成 Markdown 和 Word 报告 |
| 图表 | 加入雷达图，用于展示球员能力结构 |
| 大模型边界 | 只负责解释数据和生成报告，不负责决定评分，不允许编造新闻、伤病、转会或心理性格信息 |

## 3. 第一版明确不做的内容（系统边界）

- 不做完整网页平台、用户登录、权限系统和云端部署。
- 不训练深度学习模型，不做复杂未来转会身价预测。
- 不使用新闻、采访、社交媒体文本作为分析依据。
- 不做真实人格或心理诊断，只做「场上行为风格画像」。

> **数据来源决策（方案 B）**：原始文档设想「使用现成公开 CSV / Kaggle 数据、不做爬虫」。本项目落地时改为 **方案 B——用 [ScraperFC](https://github.com/oseymour/ScraperFC) 采集 FBref 数据**（10 联赛 × 3 赛季），以获取最新、最全、可复现的数据。采集作为**阶段 0**独立成层，产出 `data/raw/players.csv` 后，阶段 1-9 保持不变。完整实现思路见 [`F&Q/方案B-数据采集实现思路.md`](../F&Q/方案B-数据采集实现思路.md) 与 [`docs/阶段0-数据采集与位置补充.md`](./阶段0-数据采集与位置补充.md)。

## 4. 总体数据流

第一版采用本地批处理流程。**先由阶段 0 用 ScraperFC 采集 FBref 数据并产出 `data/raw/players.csv`**，之后运行主程序，系统自动完成数据清洗、筛选、评分、图表生成和报告导出。

```
ScraperFC 采集 FBref（10 联赛×3 赛季）+ Transfermarkt 主位置    ← 阶段 0（方案 B）
        ↓
   data/raw/players.csv
        ↓
Python 读取数据与字段检查                       ← 阶段 1
        ↓
数据清洗：位置统一、年龄筛选、分钟数过滤、缺失值处理、去重   ← 阶段 2
        ↓
按位置分组：Winger / CM / AM / DM               ← 阶段 2
        ↓
计算每 90 分钟效率数据与分项评分                 ← 阶段 3 / 4
        ↓
引入 Opta Power Rankings 联赛强度轻量修正        ← 阶段 5
        ↓
生成候选球员清单与 Top 20 潜力排行榜             ← 阶段 6
        ↓
为 Top 20 生成雷达图                            ← 阶段 7
        ↓
生成 Markdown 源报告                            ← 阶段 8
        ↓
导出每名球员 Word 报告与汇总报告包               ← 阶段 8 / 9
```

## 5. 技术选型

| 模块 | 推荐技术 | 作用 |
|------|---------|------|
| **数据采集（阶段 0）** | **ScraperFC** | 从 FBref 抓取球员统计、从 Transfermarkt 抓主位置（方案 B） |
| 数据读取与清洗 | pandas | 读取 CSV、处理缺失值、筛选年龄和出场时间 |
| 数值计算 | pandas + numpy | 计算 per90 数据、百分位、分项得分和总分 |
| 图表生成 | matplotlib | 生成雷达图 PNG |
| 配置管理 | YAML 或 JSON | 保存年龄阈值、位置映射、评分权重、联赛强度系数 |
| Markdown 报告 | Jinja2 或字符串模板 | 生成可检查、可追踪的报告源稿 |
| Word 报告 | python-docx | 导出正式 .docx 报告 |
| 大模型接口 | OpenAI API 或手动粘贴提示词 | 基于结构化数据生成自然语言分析 |

## 7. 核心模块职责一览

| 模块文件 | 职责 | 输入 | 输出 | 所属阶段 |
|----------|------|------|------|----------|
| `collector.py` | 方案 B：ScraperFC 采集 FBref + Transfermarkt 主位置 | 联赛/赛季列表、配置 | `data/raw/players.csv` | 阶段 0 |
| `main.py` | 串联完整流程 | CSV 路径、配置文件路径 | 所有输出文件 | 全部 |
| `config_loader.py` | 读取 YAML/JSON 配置 | config.yaml | 配置字典 | 全部 |
| `data_loader.py` | 读取 CSV 并做字段检查 | players.csv | 原始 DataFrame、字段缺失报告 | 阶段 1 |
| `cleaner.py` | 数据清洗、去重、格式统一 | 原始 DataFrame | 清洗后的 DataFrame | 阶段 2 |
| `position_mapper.py` | 统一位置名称并识别 Winger/CM/AM/DM（方案 B：优先用 Transfermarkt 主位置，FBref 粗位置兜底） | position / 主位置字段 | standard_position 字段 | 阶段 2 |
| `feature_builder.py` | 计算 per90 指标和数据完整度 | 清洗数据 | 增强字段后的 DataFrame | 阶段 3 |
| `scorer.py` | 计算分项评分和总评分 | 增强数据、权重配置 | 评分结果 | 阶段 4 |
| `league_strength.py` | 读取 Opta 联赛强度系数并轻量修正 | league 字段 | league_strength_score | 阶段 5 |
| `chart_builder.py` | 生成雷达图 | 球员分项得分 | PNG 图片 | 阶段 7 |
| `prompt_builder.py` | 生成大模型提示词 | 球员数据与评分 | prompt 文本 | 阶段 8 |
| `report_generator.py` | 生成 Markdown 报告 | 结构化球员数据 | Markdown 文件 | 阶段 8 |
| `word_exporter.py` | 生成 Word 报告 | Markdown/结构化内容、雷达图 | docx 文件 | 阶段 8 |

## 8. 配置文件设计

配置文件的作用是把「经常会调整的规则」从 Python 代码中拿出来。改最低出场时间、Top N、评分权重、位置映射、联赛强度时，不需要改主程序。

示例 `config.yaml`：

```yaml
project:
  name: young_player_scouting
filters:
  max_age: 21
  min_minutes: 600
  top_n: 20
  min_candidate_count: 100
positions:
  winger: [LW, RW, LM, RM, Winger]
  midfield: [CM, AM, DM]
scoring:
  league_strength_weight: 0.10
  base_score_weight: 0.90
  max_risk_penalty: 10
report:
  generate_markdown: true
  generate_docx: true
  generate_radar_chart: true
```

## 9. 范围失控风险（贯穿全程）

| 风险 | 表现 | 解决方案 |
|------|------|----------|
| 项目范围失控 | 想加网站、机器学习等超范围功能 | 第一版只做本地脚本闭环，后续版本再扩展 |

> 说明：方案 B 的 ScraperFC 数据采集是**经确认的数据获取方式**，不属于范围失控；但采集应保持在「产出 CSV」这一层，不扩展成实时爬虫服务或在线平台。

## 10. 最终建议

第一版 MVP 严格采用「本地 Python 脚本版」。系统的关键不是做复杂 AI，而是把数据流程跑通、把评分逻辑讲清楚、把报告生成做稳定。联赛强度统一采用 Opta Power Rankings 作为主要参考，且只作为总分中不超过 10% 的轻量修正。
