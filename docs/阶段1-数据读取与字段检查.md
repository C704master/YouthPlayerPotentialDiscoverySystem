# 阶段 1 · 数据读取与字段检查

> 对应原文档：五、核心模块设计（data_loader.py）／ 六、数据字段设计 ／ 十六、风险与解决方案（CSV 字段不统一、数据缺失）

## 阶段目标

能读取 CSV 并输出**字段检查结果**：知道有哪些字段、缺哪些字段、缺失会带来什么影响。

> **方案 B**：`data/raw/players.csv` 由 [阶段0](./阶段0-数据采集与位置补充.md) 的 ScraperFC 采集产出（FBref + Transfermarkt 主位置）。本阶段只需在 `field_mapping.yaml` 补上 FBref 的列名别名即可，无需改动读取逻辑。

## 输入 / 输出

| 项 | 内容 |
|----|------|
| 输入 | `data/raw/players.csv`（阶段 0 采集产出）、`config/field_mapping.yaml` |
| 输出 | 原始 DataFrame、字段缺失报告（控制台 + 可选文件） |
| 模块 | `data_loader.py` |

## 字段分级设计

公开 CSV 数据的字段名称往往不统一。系统应把字段分为三类，避免因为某些高级字段缺失就无法运行。

| 字段等级 | 字段示例 | 处理方式 |
|----------|----------|----------|
| **必须字段** | player_name, age 或 date_of_birth, position, team, league, minutes | 缺失时无法进入正式分析，需要在字段检查报告中提示 |
| **核心评分字段** | goals, assists, shots, key_passes, dribbles, passes, tackles, interceptions, yellow_cards, red_cards | 缺失时可以运行，但会降低数据完整度，并在报告中提示 |
| **增强字段** | xG, xA, market_value, height, weight, preferred_foot, progressive_carries, shot_creating_actions | 有则增强评分和报告解释，没有也不影响 MVP 跑通 |

## 字段映射（field_mapping.yaml）

不同数据集字段名不同（如 `goals`/`Gls`、`minutes`/`Min`）。用映射表把外部列名统一成系统内部标准名。

示例 `field_mapping.yaml`：

```yaml
player_name:
  - player_name
  - name
  - Player
minutes:
  - minutes
  - Min
  - playing_time
goals:
  - goals
  - Gls
assists:
  - assists
  - Ast
```

## 实现要点

1. 用 pandas `pd.read_csv` 读取原始数据。
2. 加载 `field_mapping.yaml`，把外部列名归一化为内部标准字段名。
3. 遍历三级字段清单，生成字段检查报告：
   - 必须字段缺失 → 明确**警告**（可能无法正式分析）。
   - 核心评分字段缺失 → 记录，后续降低数据完整度。
   - 增强字段缺失 → 记录，不阻塞。
4. 控制台打印读取行数与字段检查结果（对应验收标准）。

## 相关风险

| 风险 | 表现 | 解决方案 |
|------|------|----------|
| CSV 字段不统一 | 不同数据集字段名称不同，如 goals/Gls、minutes/Min | 使用 `field_mapping.yaml` 做字段映射 |
| 数据缺失 | 某些高级字段没有，如 xG、progressive_carries | 字段分级，缺失时降低数据完整度，不阻塞系统运行 |
